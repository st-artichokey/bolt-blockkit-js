## Feature Development

### Add post-submission confirmation message

**Planning conversation:**
- Identified three options for confirmation delivery: ephemeral message in retro channel, DM to submitter, or both
- Ephemeral: lightweight but user only sees it if viewing the retro channel
- DM (Messages tab): always visible, persistent, already has a pattern in the codebase via the "send copy" checkbox
- DM is cleaner code — reuses existing `chat.postMessage({ channel: userId })` pattern instead of introducing `chat.postEphemeral`
- User asked if Messages tab delivery was an option — confirmed it's the same as the DM approach
- User pointed out that a persistent record of every submission in Messages tab is a feature for a retro tool, not noise — serves as a personal submission history
- Decided on: always send a plain text confirmation to Messages tab after successful canvas write

**Implementation (TDD):**
1. Wrote 3 failing tests: confirmation sent on success, not sent on canvas failure, not sent when channel unconfigured
2. Added confirmation `chat.postMessage` after canvas write in `retroSubmitCallback`
3. Added early `return` on canvas write failure so neither confirmation nor DM copy is sent on error
4. Updated 2 existing tests to account for the new confirmation message count

- **Sources:**
  - [Slack chat.postMessage API](https://api.slack.com/methods/chat.postMessage)
  - Existing DM copy pattern in `listeners/views/retro-submit.js`

### Verify and clean up manifest.json

Cross-referenced all `client.*` API calls and event subscriptions in `listeners/` against Slack API docs:
- Removed unused `chat:write.public` scope — app writes canvases to the retro channel (not messages), and the bot must be in the channel for auto-discovery
- Enabled `messages_tab_enabled: true` — confirmation messages and DM copies are sent to the bot's Messages tab, so it needs to be visible in App Home
- Added missing `channels:read` scope — required for `member_joined_channel` event delivery; without it auto-channel discovery was silently broken
- Verified all remaining scopes (`canvases:write`, `channels:history`, `chat:write`) and confirmed `views.publish`/`views.open` require no scopes
- User asked to review the shortcuts section specifically — discovered the `commands` bot scope is required for global shortcuts to appear in Slack (per docs: "your app must have the `commands` permission scope")
- User asked whether `commands` should be a bot scope, user scope, or both — verified via Slack docs it's bot-only
- Added `commands` to bot scopes
